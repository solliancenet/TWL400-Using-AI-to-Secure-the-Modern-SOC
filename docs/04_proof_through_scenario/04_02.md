# Task 02 - Summarize the main failure points

## Introduction

The visualization shows *what* happened. The next piece of the Proof Through Scenario explains *why nobody stopped it*. You already named these failures during the lab — the overload diagnosis in Task 01.04 and the failure-point log from Exercise 02. This task distills them into a tight, quotable summary of the main detection and response failures, the half of the deliverable that justifies changing how the SOC operates.

## Description

You will summarize where detection and response broke down across the chain and why — signals that were present but uncorrelated and buried among the ~25 decoy incidents, including the real AI alert that fired but was never tied into the cross-layer incident — and capture it as a concise failure-point summary table.

## Success criteria

- You have produced a summary of the main detection and response failures across all five stages _(Validation: **main detection/response failures named and explained**)_
- Each failure point names both the signal that *did* exist and the reason it didn't lead to action
- The summary explicitly calls out the AI-layer failure — the real Defender for Cloud — AI threat protection alert fired, but was left uncorrelated with the poisoned grounding, the identity compromise, and the endpoint activity, and buried among the ~25 decoy incidents

### 01: Recall the failures from the investigation

Pull the failure points you already established into one place.

<summary><strong>Expand this section to view the solution</strong></summary>

1. Reopen the failure/blind-spot log from Exercise 02 and the overload diagnosis from Task 01.04. Between them you established two distinct kinds of failure:

   - **Detection failures** — a signal fired (or could have) but never rose above the noise: each stage was individually low-severity, and the stages lived in five different consoles, so no analyst saw them together.
   - **Response failures** — even where a signal was noticed, response was manual, single-domain, and far too slow to contain at the ~75-minute pace the attacker moved across layers.

2. Have Security Copilot help consolidate the per-stage failures:

   ```text

   console isolation, noise, missing correlation, etc.]

   After all five stages, add: Cross-layer gap: [one sentence naming the single

   ```

### 02: Name the AI-layer failure

Call out the gap that makes this an *AI* SOC problem, not just an alert-fatigue problem.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. State the failure plainly: the AI attack *was* detected — **Microsoft Defender for Cloud — AI threat protection** raised a real native alert on the prompt-injection probing of the **Zava Refund Agent** _(illustrative — confirm at build)_. The failure is that nothing in Zava's tooling connected that alert to the rest of the chain. It sat among the ~25 decoy incidents, uncorrelated with the poisoned grounding source, the anomalous behavior of `sp-refund-agent-inference`, the identity compromise that started it, or the endpoint execution that ended it.

2. Make the consequence explicit: because the AI alert sat disconnected from identity, infrastructure, and endpoint telemetry — and buried in the noise — the **most consequential stage was the one nobody tied together**, and the attacker reached high-value AI assets without any single tool treating it as part of one attack.

   {: .tip }
   > This is the line that separates this module from a generic alert-fatigue story. Keep it sharp: Zava's failure wasn't a missing sensor — the AI alert fired. It was the *absence of correlation across the AI layer*, with the alert lost in the ~25-decoy noise.

</details>
