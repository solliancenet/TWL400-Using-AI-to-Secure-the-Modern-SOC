# Task 02.04 — Record systemic dependencies and conflicting signals

## Introduction

The gap log from Task 02.03 says where detection failed. This final task asks a deeper question: *why was the chain able to progress at all*, and *why was it so hard to read while it was happening?* Two things made this attack work. First, **systemic dependencies** — design choices and trust relationships that let one compromise cascade into the next (a stolen session that the agent's grounding source implicitly trusted; a service principal with more reach than its job required). Second, **real-time uncertainty** — every stage looked individually benign or ambiguous, so even an analyst watching closely faced **conflicting signals** with no obvious "this is an attack" moment. Articulating both is what makes the Proof Through Scenario honest: the SOC didn't miss this because the analysts were careless, but because the system was built to let it through and to look normal while it did.

## Description

You will record the **systemic dependencies** that allowed the attack to chain across layers, and the **conflicting / uncertain signals** an analyst would have faced in real time at each stage. You'll capture both as short, quotable notes that explain the lifecycle's progression in terms of system design and human-decision difficulty — not just missing detections.

## Success Criteria

- You have recorded the systemic dependencies (trust relationships, over-privilege, and design assumptions) that let each stage enable the next
- You have recorded the conflicting/ambiguous signals an analyst faced in real time at each stage and why none of them clearly read as "attack"
- **Systemic dependencies and real-time uncertainties articulated** _(Validation: systemic dependencies and real-time uncertainties articulated)_
- These notes are saved alongside the gap log for the Exercise 04 Proof Through Scenario and the executive framing for Module 5

## Key Tasks

### 01: Map the systemic dependencies

Name the trust relationships and design choices that let the chain cascade.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Walk the lifecycle and, for each hop, ask: *what did the next stage rely on that let it succeed?* Capture the dependency, not just the event. You should land on a set like this _(illustrative — confirm at build)_:

   | Hop | Systemic dependency that enabled it |
   |-----|-------------------------------------|
   | Stage 1 → 2 | The Refund Policy Knowledge Base **implicitly trusts** authenticated uploads; a valid (stolen) session was enough to poison the grounding source — no separate review of grounding-content changes. |
   | Stage 2 → 3 | The Refund Agent **trusts its grounding source as authoritative**; poisoned content is treated as policy, so injection rides in through RAG rather than being filtered as untrusted input. |
   | Stage 3 → 4 | `sp-refund-agent-inference` is **over-privileged** relative to its task — it can reach production inference infrastructure in `rg-zava-ai-prod`, so a compromised agent context translates into infrastructure reach. |
   | Stage 4 → 5 | The endpoint **OPS-LT-0427** and the cloud identity share the same compromised user context, so cloud compromise and on-device execution reinforce each other with no segmentation between them. |
   | Across all | **Fragmented tooling (50–75 products)** is itself a systemic dependency: with no shared correlation layer, each trust boundary is policed in isolation and the cascade is invisible end to end. |

2. For each dependency, note in one phrase the design change that would break it (you don't fix it here — this seeds Module 4's operating-model and Module 5's executive recommendations). Examples: grounding-source change review; least-privilege scoping for `sp-refund-agent-inference`; treating RAG content as untrusted input.

### 02: Record the real-time conflicting signals

Capture why each stage was ambiguous in the moment.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. For each stage, write what an analyst actually saw **in real time** and why it didn't clearly read as an attack. Aim for the genuine ambiguity, not hindsight. For example _(illustrative — confirm at build)_:

   - **Stage 1 (identity):** a **medium** anonymous IP address detection — VPN and privacy-tool use is common enough that an `anonymousIpAddress` flag at medium severity is easy to triage away as a VPN user, not an attacker.
   - **Stage 2 (poisoning):** a policy document uploaded to a knowledge base **looks exactly like routine business** — vendors send policy updates; nothing about the upload itself is malicious-looking.
   - **Stage 3 (AI):** Defender for Cloud — AI threat protection **did raise an alert** on the Refund Agent probing — but as one more medium-severity item among ~25 decoys, "the agent is busy / a noisy false positive" was the easy read, and nothing tied it to the identity or infra activity to make it obviously an attack.
   - **Stage 4 (infrastructure):** a service principal named `sp-refund-agent-inference` requesting inference compute is **literally its job** — anomalous volume is the only tell, and only against a known baseline.
   - **Stage 5 (endpoint):** a script named `update_check.ps1` reads as **routine IT maintenance**; the encoded command is the giveaway, but encoded PowerShell also appears in legitimate management tooling.

2. Note the meta-point: each signal had an innocent explanation that was **more probable than "coordinated multi-stage attack"** when viewed alone. The conflict isn't signal vs. noise — it's **plausible-benign vs. plausible-malicious**, repeated five times in five consoles, with no view that aggregates the priors. That is the real-time uncertainty an AI-driven SOC has to resolve by correlating, not by lowering thresholds.

   {: .note }
   > 📷 **Screenshot to capture:** your conflicting-signals notes pairing each stage's benign explanation with its malicious tell.

</details>

### 03: Synthesize the "hard to read in real time" statement

Turn the notes into the honest narrative for the executive deliverable.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Write 2–3 quotable sentences combining both lenses: the chain progressed because **trusted relationships and over-privilege let each compromise enable the next**, and it stayed hidden because **every stage was individually more consistent with benign activity than with attack**, fragmented across five consoles. No single analyst decision was obviously wrong; the *system* let it through and made it look normal.

2. Connect it forward: this is the argument for cross-layer, machine-speed correlation — the case Exercise 03 demonstrates with a coordinated response and Exercises 04–05 carry to the CISO (Vikram), CAIO (Michelle), and the leads (Phillip, Nancy, Amy, Quinton) as the business value of an AI-driven SOC.

   {: .tip }
   > Keep this tight and non-defensive. The strongest version doesn't blame the analysts or even the tools individually — it shows that **the architecture and the alert economics made this outcome the expected one**, which is precisely what changes when correlation moves to machine speed.

</details>

You closed the reconstruction by explaining the attack at the system level: the **trust relationships and over-privilege** that let each stage enable the next, and the **conflicting, plausibly-benign signals** that made the chain hard to read while it was happening. With the systemic dependencies and real-time uncertainties articulated and saved next to your gap log and placed lifecycle, you have the full evidence base for a coordinated response.
