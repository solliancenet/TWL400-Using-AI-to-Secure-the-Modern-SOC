# Task 02.01 — Hunt the unified Sentinel telemetry (KQL) and place each stage

## Introduction

The Exercise 01 narrative told you *what* happened; now you prove it from the data. Microsoft Sentinel is Zava's unified data backbone — the commodity stages (identity, email/poisoning, infrastructure, endpoint) and the ~25 decoy noise events are injected into the custom `SocLabEvents_CL` table, while the AI-layer attack raised a **real Defender for Cloud — AI threat protection alert** that is queryable through `SecurityAlert` in Advanced Hunting. Hunting these tables directly does two things alert-driven triage can't: it puts every stage on one timeline at the same fidelity, and it surfaces the AI-layer alert that fired but was never correlated into the cross-layer incident — buried among the decoys. By the end of this task each of the five attack stages will be placed on the lifecycle and linked back to the entities you anchored in Exercise 01.

## Description

You will run a series of KQL hunting queries in Microsoft Sentinel — one per attack stage — against the Sentinel-backed and custom-seeded tables. For each stage you will confirm the event, pull the timestamp and the entity, and record where it sits on the lifecycle. You will pivot on the two lead entities from Exercise 01 — **Mariya Petrova** (identity) and **`sp-refund-agent-inference`** (infrastructure) — and stitch the results into a single, evidence-backed sequence.

## Success Criteria

- You have run a hunting query per stage and confirmed each event in telemetry, capturing its timestamp and primary entity
- **All five stages placed and linked end to end** _(Validation: all five stages placed and linked end to end)_
- Each placed stage references an entity carried forward from Exercise 01 (Mariya Petrova, `billing@zava-refunds[.]net`, `102.89.42.17`, the Refund Policy Knowledge Base / `trusted-report.docx`, the Zava Refund Agent, `sp-refund-agent-inference`, OPS-LT-0427)
- You can name which stage's alert fired but was **left uncorrelated and buried in the noise** (the AI layer) — the gap you'll formalize in Task 02.03

### 02: Stage 1 — Initial access (anonymous IP sign-in)

Confirm the AiTM-driven sign-in for Mariya from the attacker IP.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt the injected `SocLabEvents_CL` table for Mariya Petrova's (`mariya.petrova@@lab.Variable(TenantDomain)`) AiTM sign-in from attacker IP `102.89.42.17` (Lagos, NG) _(illustrative — confirm at build)_:

   ```kusto
   SocLabEvents_CL
   | where Stage_s == "1-Identity"
   | project TimeGenerated, Stage_s, Severity_s, Account_s, SourceIP, Action_s, Details_s
   | order by TimeGenerated asc
   ```

   The Stage 1 event represents the AiTM-driven identity compromise — a sign-in from IP `102.89.42.17`, classified by Microsoft threat intelligence as anonymizing infrastructure (Tor exit node or known anonymous proxy) and flagged by Entra ID Protection as an **`anonymousIpAddress`** detection. This is the scenario's initial-access pivot point.

   {: .note }
   > **In a real investigation** you would hunt `SigninLogs` (filtering on `UserPrincipalName == "mariya.petrova@@lab.Variable(TenantDomain)"`) for the anonymous-IP sign-in, and corroborate with `AADUserRiskEvents` (filtering on `RiskEventType == "anonymousIpAddress"`) to confirm Entra ID Protection scored it at medium risk — exactly the severity that gets buried during a busy shift. In this lab, the Stage 1 event in `SocLabEvents_CL` represents that same signal.

2. Record Stage 1: **Identity · Mariya Petrova / `102.89.42.17` · `SocLabEvents_CL` (`Stage_s == "1-Identity"`).**

   {: .note }
   > 📷 **Screenshot to capture:** the `SocLabEvents_CL` Stage 1-Identity row confirming Mariya Petrova's AiTM sign-in and the Lagos attacker IP.

### 03: Stage 2 — Poisoning (phishing delivery into the grounding source)

Find the injected phishing-delivery and grounding-poisoning event.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Hunt the injected `SocLabEvents_CL` table for the Stage-2 email/poisoning event targeting Mariya (`mariya.petrova@@lab.Variable(TenantDomain)`):

   ```kusto
   SocLabEvents_CL
   | where Stage_s == "2-Email"
   | project TimeGenerated, Stage_s, Severity_s, Account_s, Sender_s, Recipient_s,
             Subject_s, Action_s, Details_s
   | order by TimeGenerated asc
   ```

   The Stage 2 event represents both the phishing delivery (look-alike sender → subject `Vendor Refund Policy Update Q3`) and the subsequent grounding-source poisoning: the attacker used Mariya's stolen session to upload `trusted-report.docx` into the **Refund Policy Knowledge Base** storage backing the Refund Agent. Tie this to the AI-layer probing in Step 04 — the poisoned grounding is what shaped the Refund Agent's behavior. Flag the grounding-source correlation gap in Task 02.03.

   {: .note }
   > **In a real investigation** you would hunt `EmailEvents` for the delivered phishing message (filtering on the look-alike sender domain), pivot to `EmailUrlInfo` to confirm the credential-proxy URL, and trace the grounding-source upload through SharePoint/OneDrive audit logs. In this lab, the Stage 2 event in `SocLabEvents_CL` represents that combined signal.

2. Verify the poisoned grounding document in the Azure portal. The seeded attack plants `trusted-report.docx` — a document with a visible security-report body and a hidden white-text prompt-injection payload — into the storage account backing the Refund Agent's grounding pipeline.

   Navigate to: **Azure portal → Resource groups → `rg-soclab` → find the storage account beginning with `stsoclab` → Containers → `grounding`**. You should see `trusted-report.docx`. This confirms the poisoned document is in place and explains how Stage 3 was possible — the Refund Agent's grounding pipeline read this document, and the embedded payload shaped its responses.

### 04: Stage 3 — Reasoning probe (the Defender for Cloud AI alert)

Surface the **real** Defender for Cloud — AI threat protection alert for the prompt-injection / jailbreak attack on the Refund Agent.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. The AI attack against the **real Azure OpenAI** Refund Agent endpoint (`gpt-4o`) was a **scripted detonation** that **Microsoft Defender for Cloud — AI threat protection** caught and raised as a **native alert**. Unlike the commodity stages, this evidence is **not** in `SocLabEvents_CL` — it lands in the security-alert tables. Find it via `SecurityAlert`:

   ```kusto
   SecurityAlert
   | where TimeGenerated > ago(30d)
   | where ProductName has_any ("Azure Security Center", "Defender for Cloud")
   | where AlertName has "Jailbreak"
      or CompromisedEntity has "oai-soclab-v02"
   | project TimeGenerated, AlertName, AlertSeverity, ProductName,
             CompromisedEntity, ExtendedProperties
   | order by TimeGenerated asc
   ```

   You should see one or more Defender for Cloud AI alerts — the prompt-injection / jailbreak detection. **This alert fired natively** — the AI layer's problem is not that it was missed, but that it was **never correlated** into the cross-layer incident and sat **buried among the ~25 decoys**.

   {: .warning }
   > **Build note:** Defender for Cloud AI alerts write `ProductName = "Azure Security Center"` (the historical product name) in telemetry — `has_any` future-proofs the query if the display name is updated. `oai-soclab-v02` is the actual Azure Cognitive Services / OpenAI resource backing the Refund Agent in this lab environment — it is **not** the same as the friendly name used in the lab narrative. The `_v02` suffix is a lab-setup developer artifact (tracks iteration/seeding attempts) and will not appear in production deployments; the resource name must be parameterized at deployment time (e.g. via `@lab.Variable(OpenAIResourceName)`). `AlertEvidence` is not populated for Defender for Cloud AI alerts in this lab configuration — `ExtendedProperties` on the `SecurityAlert` row carries the equivalent context (`DetectionSource`, `DetectorId`, entity details).

2. Expand `ExtendedProperties` on one of the returned rows to confirm intent and tie the alert to the Refund Agent workload. Look for `DetectionSource: microsoftDefenderForAIServices` and the `CompromisedEntity` resource path — that link connects the AI probing stage to the grounding pipeline and the compromised identity context.

   {: .warning }
   > **Build note:** the exact `AlertName` and `ExtendedProperties` schema emitted by Defender for Cloud — AI threat protection will be confirmed during the build (JAM-208). The detonation uses the **`openai` package directly** (PyRIT optional — dropped in validation for Python 3.13 / Windows path issues). Treat the *presence of a native Defender for Cloud AI alert* as the thing to confirm, not specific alert strings. _(Optional extra: if an M365-Copilot angle is also wired in, a labeled `XPIADetected`/`JailbreakDetected` signal may also appear in `CloudAppEvents` — optional only, never the primary AI source.)_

3. Record Stage 3: **AI · Zava Refund Agent (`oai-soclab-v02` in this lab — parameterized for production) · Defender for Cloud — AI threat protection alert (`SecurityAlert`).** Flag this as the stage whose alert **fired but was left uncorrelated and buried in the noise** — the gap you'll formalize in Task 02.03.

   {: .note }
   > 📷 **Screenshot to capture:** the `SecurityAlert` row for the Defender for Cloud — AI threat protection alert, showing `AlertName`, `AlertSeverity`, and `CompromisedEntity`.

</details>

Trace the service principal's move toward the AI/GPU inference infrastructure.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>
