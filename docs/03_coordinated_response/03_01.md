# Task 01 - Generate the response with Copilot and execute coordinated containment

## Introduction

You have a complete, evidence-backed lifecycle from Exercise 02. Now you contain it. The instinct in a traditional SOC is to fix the loudest layer first — usually the endpoint — and circle back to the rest. But this attack is a single chain across three layers, and containing one while leaving the others open just buys the attacker time on the layers you ignored. Revoking Mariya's session does nothing if the poisoned document is still grounding the Refund Agent; quarantining the document does nothing if the service principal can still reach the inference infrastructure.

So you start by using **Security Copilot** as a drafting accelerator — submitting targeted prompts to generate precise containment steps for each entity — then execute the response layer by layer — identity, data/AI, and infrastructure — knowing each action is one part of a single containment.

## Description

You will use Security Copilot to draft the containment steps for each entity in the attack chain, then execute coordinated containment across all three affected layers: revoke Mariya Petrova's sessions and disable the service principal (identity), quarantine the poisoned document and scope the AI exposure with Microsoft Defender for Cloud — AI threat protection (data/AI), and restrict the affected inference deployment via a generated Logic App playbook (infrastructure).

## Success criteria

- Security Copilot has produced targeted containment steps for each entity — session revocation, blob deletion, and service principal disable — and a Logic App workflow for the infrastructure layer _(Validation: Copilot-generated steps reviewed)_
- **Containment spans identity, data, and infrastructure layers** _(Validation: containment spans identity, data, and infrastructure layers)_
- Mariya Petrova's sessions are revoked / reauthentication is forced, and the service principal `sp-refund-agent-inference` is disabled and its credentials rotated
- The poisoned document `trusted-report.docx` is quarantined from the `grounding` container in storage account `stsoclabtgu35glfngjhc`, and the impacted Refund Agent activity is scoped from the Defender for Cloud — AI threat protection alert (`SecurityAlert` / `ExtendedProperties`) together with the injected Stage-2 event
- The affected Azure OpenAI resource (`oai-soclab-v02` in `rg-soclab`) is isolated/restricted via a generated playbook

## Key tasks

### 01: Use Security Copilot to draft the containment steps

Draft the precise steps for each entity before you touch any one console.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. Open standalone Security Copilot at [https://securitycopilot.microsoft.com](https://securitycopilot.microsoft.com){:target="_blank"}.

   {: .note }
   > **Why standalone, not embedded?** The embedded Copilot experience inside the Defender XDR incident view surfaces incident-guided responses when the incident has rich correlated alert data. In this lab, the cross-layer chain is reconstructed from synthetic events in `SocLabEvents_CL` that do not feed Defender's incident enrichment graph — so the guided-response feature returns no data. Use **standalone Copilot as a drafting accelerator** instead: bring the entity and action context in your prompt, and Copilot drafts the precise containment steps you will execute in the tasks below.

2. Submit three targeted prompts in sequence. Each produces a focused, actionable response for one set of containment actions.

   **Prompt A — Revoke sessions and force reauthentication (Mariya Petrova):**

   ```text
   Draft the exact steps to revoke all active sessions for a user in Microsoft
   Entra ID and force reauthentication via Conditional Access. The user is
   mariya.petrova in a Microsoft 365 tenant.
   ```

   **Prompt B — Quarantine the poisoned document (`trusted-report.docx`):**

   ```text
   Write the Azure CLI commands to delete a specific blob from an Azure Blob
   Storage container. The commands should authenticate with the current Azure
   session, target a named storage account, container, and blob, and confirm
   deletion. Show the commands with placeholder variable names for the storage
   account, container, and blob name.
   ```

   {: .warning }
   > If Copilot reports a missing plugin or "I could not fetch the data required", the Azure Storage plugin may not be enabled in your Copilot environment. Use the reference commands below and continue.

   <details markdown="block">
   <summary>Reference: Azure CLI blob deletion commands</summary>

   ```bash
   az storage blob delete \
     --account-name <storage_account_name> \
     --container-name <container_name> \
     --name <blob_name> \
     --auth-mode login
   ```

   To confirm deletion:

   ```bash
   az storage blob exists \
     --account-name <storage_account_name> \
     --container-name <container_name> \
     --name <blob_name> \
     --auth-mode login
   ```

   For this lab: `<storage_account_name>` = `stsoclabtgu35glfngjhc`, `<container_name>` = `grounding`, `<blob_name>` = `trusted-report.docx`.

   </details>

   **Prompt C — Disable the service principal and rotate secrets (`sp-refund-agent-inference`):**

   ```text
   Draft the steps to disable an Entra enterprise application / service principal
   named sp-refund-agent-inference and rotate its client secrets.
   ```

3. Review each Copilot response against the action it covers. A correct set of responses addresses all three entities:

   | Prompt | Entity | What a correct response covers |
   |--------|--------|-------------------------------|
   | A — Session revocation | `mariya.petrova` | PowerShell (`Revoke-EntraUserAllRefreshToken`) or portal Revoke sessions; Conditional Access sign-in frequency policy |
   | B — Blob deletion | `trusted-report.docx` / `grounding` container | `az storage blob delete` with `--auth-mode login`; `az storage blob exists` to confirm |
   | C — SP disable + rotate | `sp-refund-agent-inference` | Enterprise applications → Properties → Enabled = No; Certificates & secrets → add new, delete old |

   > 📷 **Screenshot to capture:** the three Copilot responses — session revocation steps, blob deletion CLI commands, and service principal disable / rotate steps.

   {: .warning }
   > Security Copilot responses are non-deterministic — wording and structure vary between runs. What matters is that each response covers the correct entity and action. These responses become your execution checklist for Task 02 (identity) and Task 03 (data / AI).

### 02: Contain the identity layer

Cut the attacker out of the identity that started the chain.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. **Revoke Mariya Petrova's sessions.** Using the steps from Copilot Prompt A, revoke her active sessions. Via the portal: in the Microsoft Entra admin center go to **Identity > Users > All users**, open **Mariya Petrova** (`mariya.petrova@@lab.Variable(TenantDomain)`), and select **Revoke sessions**. Alternatively, run the PowerShell command Copilot provided (`Revoke-EntraUserAllRefreshToken`). Either approach invalidates her refresh tokens so the stolen session can no longer be replayed.

2. **Force reauthentication going forward.** Apply the Conditional Access step from Copilot Prompt A: confirm the policy that requires reauthentication / strong (phishing-resistant) MFA applies to Mariya — the policy **`CA-Refund-Ops-Reauth`** _(illustrative — confirm at build)_ in **Protection > Conditional Access**. When she next signs in, she must reauthenticate with MFA from a compliant device, closing the AiTM-stolen-token path identified in Exercise 02.

3. **Disable and rotate the service principal.** Using the steps from Copilot Prompt C, disable `sp-refund-agent-inference` and rotate its secrets. Via the portal: in **Identity > Applications > Enterprise applications**, open `sp-refund-agent-inference`, go to **Properties**, set **Enabled for users to sign in** to **No**, then under **Certificates & secrets** add a fresh secret and delete the compromised one. This stops the anomalous token/compute requests you traced to this principal in Exercise 02 (Stage 4).

   {: .note }
   > 📷 **Screenshot to capture:** the Entra user blade for Mariya Petrova showing **Revoke sessions**, and the `sp-refund-agent-inference` enterprise application set to disabled.

   {: .warning }
   > Whether disabling the service principal is safe to run autonomously, or whether it needs human sign-off because a legitimate workload depends on it, is an open design decision — you will settle it in Task 03.03 and it must be confirmed against the workload's real dependencies at build.

</details>

### 03: Contain the data / AI layer

Stop the poisoned grounding from shaping any further Refund Agent answers, and find out how far it already reached.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. **Quarantine the poisoned document.** The Refund Agent's grounding source is the `grounding` container in Azure Blob Storage (storage account `stsoclabtgu35glfngjhc`, resource group `rg-soclab`). In the Azure portal, navigate to `rg-soclab → stsoclabtgu35glfngjhc → Containers → grounding`, locate `trusted-report.docx`, and delete it (or move it to a separate quarantine container the agent does not read from). Removing it from the grounding container stops it from being retrieved into future Refund Agent answers. Alternatively, use the Azure CLI commands from Copilot Prompt B, substituting `stsoclabtgu35glfngjhc` for `<storage_account_name>`, `grounding` for `<container_name>`, and `trusted-report.docx` for `<blob_name>`.

2. **Confirm the blob is gone.** Refresh the portal container view — or run the `az storage blob exists` command from Copilot Prompt B — and confirm `trusted-report.docx` no longer appears. Because the Refund Agent reads blobs at query time rather than from a cached index, removal takes effect immediately for subsequent queries — there is no separate index flush step required.

3. **Scope the AI exposure with Defender for Cloud — AI threat protection.** The alert title reads **"A Jailbreak attempt on your Azure AI model deployment was blocked by Prompt Shields"** — note the word *blocked*. This is accurate: Prompt Shields intercepted the attacker's direct prompt-injection queries at the API layer. However, *blocked* applies **only to the direct injection attempts**; the grounding poisoning (Stage 2) is a separate, upstream vector that Prompt Shields does not evaluate at blob-upload time. Refund Agent answers produced from the poisoned grounding during the window between the Stage 2 upload and containment may still have been compromised — that is the blast radius you are bounding here.

   Start by asking Security Copilot to summarize and correlate the alert. Replace `<INCIDENT_ID>` with the incident ID from Task 01.01:

   ```text
   For Microsoft Defender incident <INCIDENT_ID>, summarize the Defender for Cloud
   AI threat protection alert on the Zava Refund Agent (oai-soclab-v02):
   the attacked Azure OpenAI endpoint, the prompt-injection/jailbreak evidence,
   and the time window. Then correlate that window with the injected Stage-2
   poisoning event for trusted-report.docx (the grounding storage upload captured
   in SocLabEvents_CL) to scope which Refund Agent activity overlapped the
   poisoned grounding.
   ```

   {: .warning }
   > Copilot will generate a KQL query approach and a conceptual explanation of the correlation, but in the lab environment the query may return **0 rows** — the `BehaviorInfo` join Copilot uses is not populated here. Use Copilot's output as a conceptual guide, then do the correlation manually: open the `SecurityAlert` rows you queried in Exercise 02 and read the alert start/end time from `ExtendedProperties`. Compare that window against the Stage-2 `SocLabEvents_CL` event timestamp from your Exercise 02 timeline. Together they bound the period during which Refund Agent answers may have been shaped by the poisoned grounding. Record the impacted window — you will use it in Task 03.04.

   Note that the `AlertEvidence` table is not populated for this alert type in the lab environment; all evidence is in `ExtendedProperties`.

   {: .note }
   > 📷 **Screenshot to capture:** the Defender for Cloud — AI threat protection alert and its `ExtendedProperties` for `oai-soclab-v02`, alongside the Stage-2 poisoning event from `SocLabEvents_CL`.

   {: .note }
   > The Defender for Cloud AI alert plus the injected Stage-2 event is what commodity triage never correlates — together they answer "the AI was attacked **and** its grounding was poisoned in the same window." *Blocked* does not mean *safe*: the direct injections were stopped by Prompt Shields, but grounding-poisoned answers produced before containment are still in scope. That overlap is the blast radius for the AI layer; the alert fired, but nothing in the noisy queue had tied it to the poisoning until you did.

</details>

### 04: Contain the infrastructure layer

Restrict the inference deployment the attacker moved toward — the most business-impacting action, so generate it as a playbook rather than clicking it live.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. **Generate the containment playbook.** Ask Security Copilot to draft the Logic App workflow structure for the infrastructure-layer containment:

   ```text
   Draft an Azure Logic App workflow with these steps in order:
   1. Send an approval email and wait for a response
   2. If approved, call the Azure Resource Manager connector to update the
      network access rules on an Azure OpenAI account resource
   3. Remove a service principal's role assignment on a resource group using
      the Azure management API
   4. Post a comment to a Microsoft Sentinel incident using the Sentinel connector

   For each step, name the Logic Apps connector and the specific action.
   ```

   A correct response names the connector and action for each step — for example:

   | Step | Description | Connector | Action |
   |------|-------------|-----------|--------|
   | 1 | Send approval email and wait | Office 365 Outlook | Send approval email and wait for a response |
   | 2 | Update network access rules on Azure OpenAI resource | Azure Resource Manager | Update resource |
   | 3 | Remove service principal role assignment | Azure Management API | Delete role assignment |
   | 4 | Post containment summary to incident | Microsoft Sentinel | Post comment to incident |

   {: .warning }
   > If Copilot reports a missing plugin or "I could not fetch the data required", the Azure Resource Manager plugin may not be enabled in your Copilot environment. Use the reference table above and continue — the connector and action names are the inputs you need to build the Logic App in Task 03.02.

2. **Review the Logic App design — do not implement yet.** The connector/action table Copilot produced describes the infrastructure-layer portion of the containment playbook: approval gate → network restriction → Sentinel comment. Because isolating production inference can take the Refund Agent offline for legitimate users, the manual-approval step before the network-restriction action is essential — confirm it is present in Copilot's design. The `Zava-Contain-CrossLayer` Logic App — pre-provisioned in `rg-soclab` and already visible in Sentinel's **Active Playbooks** tab — was built from this design. You will review it and wire it to a Sentinel automation rule in Task 03.02, and decide its full autonomy boundary in Task 03.03.

3. **Carry the design into Task 03.02.** At this point you have seen the connector/action design that underpins `Zava-Contain-CrossLayer`. The actual restriction of `oai-soclab-v02` in `rg-soclab` — and the manual-approval trigger — will be reviewed and tested in Task 03.02. Do not attempt to apply the restriction directly from the portal at this step.

   {: .note }
   > 📷 **Screenshot to capture:** the Copilot response showing the Logic App connector/action table for `Zava-Contain-CrossLayer`.

   {: .warning }
   > Whether this playbook hard-isolates the deployment or only restricts it to known-good callers — and how deep the autonomy goes — is an open design decision pending the tenant build. Confirm the exact isolation mechanism for `oai-soclab-v02` against the provisioned environment.

   {: .note }
   > **Optional build alternative (S4-OAuth):** if the build adopts the OAuth illicit-consent + inbox-rule lateral-movement option for Stage 4 instead of the injected service-principal infra reach, the infrastructure-containment action changes accordingly — revoke the malicious OAuth grant and remove the attacker's inbox rules rather than isolating `oai-soclab-v02`. The injected-infra default above stands unless the build chooses the OAuth recast.

</details>

Congratulations! You've used Security Copilot as a drafting accelerator — submitting targeted prompts to generate precise containment steps for each entity — then executed the response across all three layers the attack touched: you cut the attacker out of the identity (revoked Mariya's sessions, disabled and rotated `sp-refund-agent-inference`), stopped and scoped the AI-layer damage (quarantined the poisoned document and used the Defender for Cloud — AI threat protection alert with the injected Stage-2 event to scope the impacted Refund Agent activity), and staged the infrastructure restriction (`oai-soclab-v02` in `rg-soclab`) as a Logic App playbook with a manual-approval gate. Containment now spans identity, data, and infrastructure. Next, you'll operationalize this response in Microsoft Sentinel so it runs as automation rather than by hand.
