# Task 03.03 — Define the autonomous-vs-human-approved boundary

## Introduction

The automation you configured in Task 03.02 already made one decision for you: it ran the identity and data/AI containment immediately and waited for a human before isolating the inference infrastructure. That split was not arbitrary, and it is the single most important governance choice an AI-driven SOC makes. Run too little autonomously and you are back to 75-minute manual response. Run too much autonomously and one false positive can take a production AI service offline on its own authority.

In this task you make that boundary **explicit and defensible**. You classify each response action as autonomous or human-approved, write the rationale, and tie it back to the personas who own the risk — because when Vikram (CISO) or Quinton (Infrastructure & Architecture Lead) asks "why did the machine do that without asking?", the answer has to already exist.

## Description

You will define which response actions from this exercise may run autonomously versus which require human approval, record the rationale for each, and note how the `Zava-Contain-CrossLayer` automation — reviewed in Task 03.02 — reflects that boundary.

## Success Criteria

- **Autonomous-vs-human actions are defined with rationale** _(Validation: autonomous-vs-human actions defined with rationale)_
- Each response action is classified as **autonomous** or **human-approved**, with a one-line rationale
- The classification is grounded in two axes: **reversibility** and **business impact (blast radius)**
- The `Zava-Contain-CrossLayer` playbook's approval placement matches the boundary you define

## Key Tasks

### 01: Classify each action against reversibility and blast radius

Decide what the machine may do on its own.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Take the response actions from Task 03.01 and score each on two axes:
   - **Reversibility** — if this fires on a false positive, how easily is it undone?
   - **Business impact / blast radius** — who and what is affected if it fires when it shouldn't?

2. Actions that are **easily reversible and narrowly scoped** are good candidates to run autonomously; actions that are **disruptive or hard to reverse** should wait for a human. Build the decision table:

   | Response action | Layer | Mode _(illustrative)_ | Rationale |
   |-----------------|-------|------------------------|-----------|
   | Revoke Mariya Petrova's sign-in sessions | Identity | **Autonomous** | Cheap and reversible — she simply reauthenticates with MFA. Stops token replay immediately; near-zero blast radius. |
   | Apply reauthentication CA policy (`CA-Refund-Ops-Reauth`) | Identity | **Autonomous** | Narrowly scoped to one user; easily reversed by removing or narrowing the policy. Closes the AiTM-stolen-token path without affecting any other user. _(Executed manually in Task 03.01; not a step in `Zava-Contain-CrossLayer`.)_ |
   | Disable the service principal `sp-refund-agent-inference` | Identity | **Autonomous** | Reversible (re-enable). The principal is already behaving anomalously; disabling it stops lateral movement without taking the user-facing agent down. |
   | Quarantine `trusted-report.docx` from the `grounding` container | Data / AI | **Autonomous** | Reversible (restore from quarantine). Removing one poisoned doc from grounding does not disrupt the service; high upside in stopping further poisoned answers. |
   | Isolate / restrict the Azure OpenAI resource `oai-soclab-v02` | Infrastructure | **Human-approved** | Business-impacting — can take the Refund Agent offline for legitimate staff. Needs an owner's judgment that containment outweighs the outage. |
   | Disable the Zava Refund Agent entirely | AI / App | **Human-approved** | Highest blast radius — full loss of the service. Reserved for an explicit human decision. _(Not implemented in `Zava-Contain-CrossLayer` — this is a hypothetical escalation action beyond the current playbook scope.)_ |

   {: .note }
   > **Optional build alternative (S4-OAuth):** if the build uses the OAuth illicit-consent + inbox-rule option for Stage 4, the infrastructure-isolation row above is replaced by **revoke the malicious OAuth grant + remove the attacker's inbox rules** — both reversible and narrowly scoped, so they are good **autonomous** candidates. The injected-infra default (human-approved isolation) stands unless the build chooses the OAuth recast.

3. Capture the boundary as a single rule of thumb you can defend: *autonomous where the action is reversible and low-blast-radius (revoking sign-in sessions, applying a narrowly scoped CA policy, quarantining the poisoned document, disabling an anomalous service principal); human-approved where it is business-impacting (isolating production inference infrastructure, disabling the Refund Agent).*

### 02: Make the boundary accountable

Record who owns each gate and capture the rationale.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. **Assign a named approver** to each human-approved action. Using the personas from this scenario:
   - Isolating `oai-soclab-v02` → **Quinton** (Infrastructure & Architecture Lead), with **Vikram** (CISO) notified.
   - Disabling the Refund Agent entirely (hypothetical escalation) → **Amy** (App/AI Security Lead), with CISO sign-off.

   These are illustrative role assignments grounded in the scenario personas. In a real deployment, confirm the approver for each gated action with the organization's actual stakeholders before operationalizing the playbook.

2. **Write the boundary as a single defensible rule** — one sentence you could read aloud to Vikram in a 30-second hallway conversation:

   > *Automate the actions that buy time and are cheap to undo; require a human for the actions that cost the business and are hard to undo.*

   That sentence is the autonomy boundary. If anyone asks why the machine revoking sessions without asking is different from the machine isolating the deployment without asking, the answer is already in that sentence.

3. **Draft the governance rationale** in your running notes. Write the artifact you would hand to a CISO: for each autonomous action, name it and state in one line why it is safe to automate; for each human-approved action, name it and state in one line what the gate is protecting against. This is the governance artifact Module 4 turns into the SOC operating model.

## Summary

You turned an implicit automation choice into an explicit, defensible boundary: revoking sign-in sessions, applying the reauthentication CA policy, quarantining the poisoned document, and disabling the anomalous service principal run autonomously because they are reversible and low-blast-radius, while isolating production inference infrastructure and disabling the Refund Agent require named human approval because they are business-impacting. Each call has a named rationale and an owner, and the `Zava-Contain-CrossLayer` playbook reflects the boundary. Next, you'll document the risk that remains even after this containment.
