# Task 02 - Operationalize the response with Sentinel automation

## Introduction

In Task 03.01 you used Security Copilot to draft the containment steps for each entity, then executed them manually across three layers. That works once, with you driving — but the whole point of an AI-driven SOC is that this same response fires **at machine speed** the next time the chain appears, without waiting for an analyst to be awake and looking. Roughly 75 minutes of attacker dwell time in this scenario is, in large part, the cost of manual, console-by-console response.

In this task you operationalize the response: you review the pre-provisioned **`Zava-Contain-CrossLayer`** Logic App playbook — whose design was informed by the connector/action table Copilot produced in Task 03.01 and extended to incorporate the identity and data/AI steps from Tasks 02 and 03 — wire it to **Microsoft Sentinel automation** through an automation rule, and confirm it performs the coordinated cross-layer containment with the human-approval gate preserved where it matters.

## Description

You will review the pre-provisioned `Zava-Contain-CrossLayer` Logic App playbook, configure a Microsoft Sentinel automation rule that runs it when this multi-stage incident is created or updated, and confirm the playbook performs the coordinated cross-layer containment.

## Success criteria

- **Sentinel automation is configured/triggered for the response** _(Validation: Sentinel automation configured/triggered for the response)_
- An automation rule binds the multi-stage incident to the `Zava-Contain-CrossLayer` playbook
- A run of the playbook is observed completing end-to-end (with the manual-approval step honored for the business-impacting action; Steps 1–2 run as Compose stubs in this lab environment)
- The incident records the automation/playbook activity in its activity log

Locate the pre-provisioned Logic App **`Zava-Contain-CrossLayer`**. This playbook was built from the connector/action design Copilot produced in Task 03.01, extended to incorporate the identity and data/AI containment steps you executed manually in Tasks 02 and 03 of that exercise.

2. Open it in the Logic Apps designer and verify it contains the following steps, matching the Copilot design:

   | Step | Layer | Connector | Action |
   |------|-------|-----------|--------|
   | 1 | Identity | Microsoft Entra | Revoke sign-in sessions (Mariya Petrova) |
   | 2 | Identity | Microsoft Entra | Disable service principal `sp-refund-agent-inference` |
   | 3 | Data / AI | HTTP (MSI auth) | Delete blob `trusted-report.docx` from `grounding` container |
   | 4 | — | Office 365 Outlook | **Manual approval** — send approval email and wait for response |
   | 4b | — | (If condition) | Evaluate approval response: if **Approved** → run Step 5; if **Denied** → skip to Step 6 |
   | 5 | Infrastructure | HTTP (MSI auth) | Update network access rules on `oai-soclab-v02` _(inside If-True branch of Step 4b)_ |
   | 6 | — | Microsoft Sentinel | Post containment summary comment to the incident _(runs regardless of approval outcome; message text reflects the decision)_ |

   {: .note }
   > **How this maps to the Copilot design from Task 03.01:** Copilot produced a 4-step infrastructure-layer workflow (approval → network restriction → SP role removal → Sentinel comment). The full `Zava-Contain-CrossLayer` Logic App prepends the identity and data/AI steps (Steps 1–3 above) — the same actions you ran manually in Tasks 02 and 03 of Task 03.01. The SP role-removal step from Copilot's design does not appear as a separate Logic App step; SP containment is handled by the disable action in Step 2.

   {: .note }
   > **Designer implementation notes:** Steps 3 and 5 use **HTTP actions with Managed Identity** authentication rather than named Azure Blob Storage or Azure Resource Manager connector cards — they call the same underlying REST APIs but appear as "HTTP" steps in the designer. Step 4b is a separate **If** condition step that evaluates the approval response from Step 4; Step 5 lives inside Step 4b's True branch and is only visible when you expand it.

   {: .warning }
   > **Lab constraint — Steps 1 & 2 appear as Compose stubs:** The Microsoft Graph API permissions required for live session-revocation and service-principal disable calls could not be provisioned in this tenant. In the Logic App designer, Steps 1 and 2 will appear as **Compose** (pass-through) actions rather than Microsoft Entra connector calls — this is expected and by design. The containment those steps represent was already executed manually in Task 03.01. Steps 3–6 run live.

3. Confirm the **manual-approval gate** is at Step 4/4b — before the infrastructure isolation — so Steps 1–3 run immediately while the business-impacting Step 5 waits for human approval inside the If-True branch.

### 02: Create the automation rule

Bind the playbook to the incident so it runs without a human kicking it off.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. In **Configuration > Automation**, select **Create > Automation rule**.

2. Configure the rule **`AR-Contain-CrossLayer`** _(illustrative — confirm at build)_:

   - **Trigger:** *When incident is created* (and, optionally, *When incident is updated*).
   - **Conditions:** scope to this attack. Use **Title** contains a stable substring from the actual title of your incident (the title is system-generated — use the one you noted in Task 01.01, not a hardcoded string). Alternatively, match on **Tactics** includes lateral movement or credential access, or **Severity** equals High. Keep the condition tight so the playbook only fires on this cross-layer pattern, not on every incident.
   - **Actions:** *Run playbook* → select **`Zava-Contain-CrossLayer`**.

3. Confirm Sentinel has permission to run the playbook (the Logic App's managed identity / the Sentinel automation account must be granted the **Microsoft Sentinel Automation Contributor** role on the playbook's resource group). Save the rule.

   {: .note }
   > 📷 **Screenshot to capture:** the automation rule editor showing the trigger, the incident-title condition, and the **Run playbook** action pointing at `Zava-Contain-CrossLayer`.

</details>

### 03: Trigger and verify

Watch the response run as automation.

<details markdown="block">
<summary><strong>Expand this section to view the solution</strong></summary>

1. Trigger the automation. From the incident's **Actions > Run playbook** menu, manually trigger `Zava-Contain-CrossLayer` against the existing incident. When the playbook reaches the **manual-approval** step, approve the infrastructure isolation. The approval email is sent to `mariya.petrova@AcesaDev.onmicrosoft.com` via the `office365-zava` connection — open [Outlook Web Access](https://outlook.office.com){:target="_blank"} signed in as Mariya Petrova, locate the approval email from the Logic App, and click **Approve**. _(In a production deployment the email would go to the named infrastructure approver — Quinton, per Task 03.03. In this lab, Mariya's is the only available approval mailbox.)_

   {: .note }
   > **Steps 1 & 2 (identity containment)** run as acknowledged Compose steps in this lab environment — they emit a status string confirming the actions were completed manually in Task 03.01, rather than calling Microsoft Graph directly. This is by design: assigning Microsoft Graph API permissions to a managed identity in a shared lab tenant requires Entra ID replication that can exceed acceptable setup time. The containment pattern — what those steps do, why they run before the approval gate, and how they fit the cross-layer response — is what matters for this exercise. **Step 3** (blob delete) executes live against storage; it completes gracefully whether `trusted-report.docx` is present or already absent.

2. Verify the run. In the playbook's **Run history** (Logic Apps) confirm the run succeeded end-to-end, and on the incident's **Activity log** confirm `Zava-Contain-CrossLayer` posted its containment summary comment.

   {: .note }
   > 📷 **Screenshot to capture:** the playbook run history showing a successful run, and the incident activity log entry posted by `Zava-Contain-CrossLayer`.

   {: .tip }
   > This is the difference Module 4 builds on: the same coordinated containment you ran by hand in Task 03.01 now executes from a single incident trigger, in seconds, with one human approval instead of multiple manual console visits.

</details>
