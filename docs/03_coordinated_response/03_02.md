# Task 03.02 — Operationalize the response with Sentinel automation

## Introduction

In Task 03.01 you used Security Copilot to draft the containment steps for each entity, then executed them manually across three layers. That works once, with you driving — but the whole point of an AI-driven SOC is that this same response fires **at machine speed** the next time the chain appears, without waiting for an analyst to be awake and looking. Roughly 75 minutes of attacker dwell time in this scenario is, in large part, the cost of manual, console-by-console response.

In this task you operationalize the response: you review the pre-provisioned **`Zava-Contain-CrossLayer`** Logic App playbook — whose design corresponds to the connector/action table Copilot produced in Task 03.01 — wire it to **Microsoft Sentinel automation** through an automation rule, and confirm it performs the coordinated cross-layer containment with the human-approval gate preserved where it matters.

## Description

You will review the pre-provisioned `Zava-Contain-CrossLayer` Logic App playbook, configure a Microsoft Sentinel automation rule that runs it when this multi-stage incident is created or updated, and confirm the playbook performs the coordinated cross-layer containment.

## Success Criteria

- **Sentinel automation is configured/triggered for the response** _(Validation: Sentinel automation configured/triggered for the response)_
- An automation rule binds the multi-stage incident to the `Zava-Contain-CrossLayer` playbook
- A run of the playbook is observed completing its containment actions (with the manual-approval step honored for the business-impacting action)
- The incident records the automation/playbook activity in its activity log

## Key Tasks

### 01: Review the containment playbook

Confirm what you are about to operationalize.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. In the Microsoft Sentinel (or unified SecOps) portal, go to **Configuration > Automation** and open the **Active playbooks** tab. Locate the pre-provisioned Logic App **`Zava-Contain-CrossLayer`**. This playbook was built from the connector/action design Copilot produced in Task 03.01.

2. Open it in the Logic Apps designer and verify it contains the following steps, matching the Copilot design:

   | Step | Layer | Connector | Action |
   |------|-------|-----------|--------|
   | 1 | Identity | Microsoft Entra | Revoke sign-in sessions (Mariya Petrova) |
   | 2 | Identity | Microsoft Entra | Disable service principal `sp-refund-agent-inference` |
   | 3 | Data / AI | Azure Blob Storage | Delete blob `trusted-report.docx` from `grounding` container |
   | 4 | — | Office 365 Outlook | **Manual approval** — send approval email and wait for response |
   | 5 | Infrastructure | Azure Resource Manager | Update network access rules on `oai-soclab-v02` |
   | 6 | — | Microsoft Sentinel | Post containment summary comment to the incident |

3. Confirm the **manual-approval step** sits at Step 4 — before the infrastructure isolation — so Steps 1–3 run immediately while the business-impacting Step 5 waits for a human.

   {: .note }
   > 📷 **Screenshot to capture:** the `Zava-Contain-CrossLayer` Logic App in the designer, showing the layered containment steps and the manual-approval gate before the isolate-deployment action.

   {: .warning }
   > The split between the auto-run steps and the approval-gated step encodes the autonomy boundary you define in Task 03.03. If the build resolves that boundary differently, update the playbook's approval placement to match — the canon for that decision lives in Task 03.03.

</details>

### 02: Create the automation rule

Bind the playbook to the incident so it runs without a human kicking it off.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. In **Configuration > Automation**, select **Create > Automation rule**.

2. Configure the rule **`AR-Contain-CrossLayer`** _(illustrative — confirm at build)_:

   - **Trigger:** *When incident is created* (and, optionally, *When incident is updated*).
   - **Conditions:** scope to this attack. Use **Title** contains a stable substring from the actual title of your incident (the title is system-generated — use the one you noted in Task 01.01, not a hardcoded string). Alternatively, match on **Tactics** includes lateral movement or credential access, or **Severity** equals High. Keep the condition tight so the playbook only fires on this cross-layer pattern, not on every incident.
   - **Actions:** *Run playbook* → select **`Zava-Contain-CrossLayer`**.

3. Confirm Sentinel has permission to run the playbook (the Logic App's managed identity / the Sentinel automation account must be granted the **Microsoft Sentinel Automation Contributor** role on the playbook's resource group). Save the rule.

   {: .note }
   > 📷 **Screenshot to capture:** the automation rule editor showing the trigger, the incident-title condition, and the **Run playbook** action pointing at `Zava-Contain-CrossLayer`.

</details>

### 03: Trigger and verify

Watch the response run as automation.

<details markdown="block">
<summary><strong>Expand this section for detailed steps</strong></summary>

1. Trigger the automation. From the incident's **Actions > Run playbook** menu, manually trigger `Zava-Contain-CrossLayer` against the existing incident. When the playbook reaches the **manual-approval** step, approve the infrastructure isolation.

   {: .note }
   > The identity and data/AI steps (1–3) may show as already-applied — Mariya's sessions were revoked and `trusted-report.docx` was deleted manually in Task 03.01. This is expected. The playbook run demonstrates that the same automation would fire at incident-creation speed if triggered before any manual action. Session revocation and SP disable are idempotent; the blob-delete step will complete gracefully if the blob is already absent.

2. Verify the run. In the playbook's **Run history** (Logic Apps) confirm the run succeeded end-to-end, and on the incident's **Activity log** confirm `Zava-Contain-CrossLayer` posted its containment summary comment.

   {: .note }
   > 📷 **Screenshot to capture:** the playbook run history showing a successful run, and the incident activity log entry posted by `Zava-Contain-CrossLayer`.

   {: .tip }
   > This is the difference Module 4 builds on: the same coordinated containment you ran by hand in Task 03.01 now executes from a single incident trigger, in seconds, with one human approval instead of multiple manual console visits.

</details>

## Summary

You operationalized the coordinated response: the pre-provisioned `Zava-Contain-CrossLayer` Logic App — whose design matches the Copilot connector/action table from Task 03.01 — now runs from a Sentinel automation rule bound to the multi-stage incident, executing the identity and data/AI containment autonomously and gating the business-impacting infrastructure isolation behind a human approval. The response is no longer a manual drill — it is automation that fires at machine speed. Next, you'll make the autonomy split explicit: which actions may run with no human, which require approval, and why.
